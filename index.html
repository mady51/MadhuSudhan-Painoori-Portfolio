The system, at a glance

How the agent thinks

A single SDET run: a DB-backed worker queue claims the job, gathers context, embeds and retrieves across seven lenses, reasons with Bedrock across two concurrency lanes, and stops at a human approval gate before any reversible write.

  1. Trigger
    POST /api/sdet-agent/runseditor · { workspace, story_key, tester_prompt? } — queued, no Redis or Celery.
  2. Queue
    DB-backed worker poolN consumers claim atomically (FOR UPDATE SKIP LOCKED), with per-workspace fairness caps, retry/backoff, and trigger dedupe.
  3. Context
    Jira story + PR / dev-statusSummary, components, links, fixVersions, and changed-file counts from GitHub — plus optional Rovo AI enrichment.
  4. Retrieval
    Titan embedding → pgvector (HNSW)7 lenses: candidate TestSigma cases · KB pages · code symbols · historical defects · Salesforce metadata · integration touchpoints · user docs.
  5. Analysis
    AWS Bedrock — ClaudeTwo concurrency lanes (interactive + background) classify each case REUSE / UPDATE / NEW with reasoning, lens evidence, confidence, and a parallel performance-impact assessment.
  6. Approval
    Human-in-the-loop gateEditors review and edit; superadmins approve behind an 85% similarity guard on NEW cases. Nothing is written without sign-off.
  7. Push
    Reversible TestSigma writeNEW → create, UPDATE → update, REUSE → no-op. Every push stores a before/after snapshot, so any write can be rolled back.

7-Lens RAG Retrieval

A query embedding fans out across candidate test cases, KB pages, code symbols, historical defects, Salesforce metadata, integration touchpoints, and user docs — so proposals are grounded in real context, not guesses.

Vector Search at the Core

1024-dim Titan embeddings indexed with pgvector HNSW cosine search, living right beside the relational data in PostgreSQL — no separate vector store to operate.

Dual-Lane Bedrock Concurrency

An interactive lane keeps on-demand runs responsive while a capped background lane handles bulk revamp and release analysis — so heavy jobs never starve the UI.

Human-in-the-Loop, Reversible

Editors review, superadmins approve behind an 85% similarity guard, and every push to TestSigma stores a before/after snapshot — so any write can be rolled back.

Self-Healing Automation

Generates Playwright specs via the Claude Code CLI, validates against the live app, and self-heals up to N times — a real product failure is classified and surfaced, not healed forever.

Built to Run Anywhere

One multi-stage Docker image (non-root UID 10001, dropped capabilities), Postgres on an internal network, additive Alembic migrations, and optional OIDC SSO with secrets encrypted at rest.
